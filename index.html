Introduction/overview

Recently I have been building a rudimentary record system for use by an education provider. In this article I will detail the process and methods I used for each class.

Architecture and classes

In my classes I have used inheritance to create my class structure which is the act of using a super class (the class a subclass is inheriting from) as a template where the subclass (the class that is inheriting from the super class) then has the same behaviours as the super class but can add or override them. An example of a basic inheritance diagram is shown in the diagram below from the official Java documentation. (What Is Inheritance? (The Java™ Tutorials > Learning the Java Language > Object-Oriented Programming Concepts) (oracle.com))

With the use of classes, I have used encapsulation to increase the security of my code. Encapsulation is the use of the private keyword, getters, and setters to restrict direct access to variables. The advantage of this that I have taken advantage of in my program is that it provides data hiding, protecting the variables from external sources but other advantages that are useful but not applicable in my program include increased control and reduced exceptions of the variables (OOP Series: Encapsulation In Java - DEV Community)

Main class

The main class is where the basic initialization of key properties and some generic methods takes place.

The main method is called when the program runs and it’s where the Staff, Student, and Course lists are populated using functions in the ReadFiles class. An instance of the Menu class called menu is also instantiated and then the mainMenu function of the Menu class is run.

Globals class

The Globals class contains a few variables that can be used anywhere in the program, this is done primarily to circumvent the lack of a dedicated database for the system and allows the program to access all stored data easily. These are public static so can be accessed from anywhere in the program.

The scanner is created in the Globals class, parsing the values it is given. The scanner takes System.in which means it parses the user’s input and allows them to interact with the program. You can read more about the Scanner class on the Oracle official docs. This uses the final keyword which means after a value is declared, it cannot be modified elsewhere (final Keyword in Java - GeeksforGeeks), adding security as it should not be changed.

Three ArrayList objects are created that store staff (StaffList), students (StudentList) and courses (CourseList) that have been recorded. These are ArrayList objects as opposed to a regular array as they must have a dynamic length as the program allows the creation of new staff, students, and courses and a regular array has a static length and cannot be changed after initialisation (https://howtodoinjava.com/java/array/array-vs-arraylist/).

Menu class

The menu class has all the generic menu functions. Most menus with user interaction are done using while loops to repeat the menu until it is not needed anymore and selection for acting upon user input. For menus I have used primarily switch-case statements as they are faster for selection statements with more than 5 cases (switch vs if else - GeeksforGeeks) which will not affect these menus now but allows for expansion without having to refactor for efficiency.

The login function is also part of this class which uses a for loop to check through each student or staff member and makes a comparison between the id input and the object in that iteration of the loop to check if they match. If it does match, the user is prompted to enter the password, which uses a while loop and an int counter to allow a certain number of attempts before being returned to the main menu. To check the passwords, the program uses and if statement and .equals() to perform a string comparison. String.equals() compares the content of two String objects, it is important to use this instead of == because a String is not a primitive type, it is an object so == compares the references of the strings which may be different even if the content of the String is equal (Java String.equals() with Examples - HowToDoInJava).

Person class

The person class is an abstract class used as a base class for system user classes to inherit from and holds the basic properties and methods that are used by all users. This is an abstract class which means that objects cannot be created off it and to create an object, another class must inherit it and be used to create the object provided the class inheriting it isn’t abstract (Java Abstraction (w3schools.com)). I’ve done this because there shouldn’t be a generic person made and it provides security by hiding information from the user that isn’t essential to them.

Staff class

The staff class inherits from the person class and provides the basic functionality of a staff member such as managing courses attached to themselves. It declares the mStaffCourseList ArrayList which contains all the courses attached to the staff member.

Admin class

The admin class inherits from the staff class and adds functionality for only admins such as adding new staff members. It also overrides the actionMenu() method which is used to choose the action the user would like to make to allow the user to access the functions available to only them. This is done using method overriding which means the actionMenu() function declared in the staff class, the base class, is overridden and the function in the admin class, the subclass, is used (Overriding and Hiding Methods (The Java™ Tutorials > Learning the Java Language > Interfaces and Inheritance) (oracle.com)).

Teacher and student classes

The teacher and student classes inherit from the staff and person classes respectively and adds functionality only available to their class type and overrides the actionMenu() method.

Courses

The course class is a basic class that creates a structure for courses to be created, using the CourseTypes enum to determine the type of course from a static set of values.

CourseTypes

The CourseTypes is an enum which holds each type of course as constants. An enum is appropriate for this as there is a limited number of CourseTypes available in the program, making it easily readable and preventing any errors caused by passing in an invalid value (java - What are enums and why are they useful? - Stack Overflow).

WriteFiles

The WriteFiles class contains static methods to write objects to files such as staff, students, and courses. It creates a new file with the file path being set to the directory passed into the function and the id of the object appended to it. It then creates the directory if it doesn’t already exist using mkdir(). I then created a FileWriter passing in the File created before and use that to create a new BufferedWriter. Using BufferedWriter allows efficient writing (Java.io.BufferedWriter class methods in Java - GeeksforGeeks) by making less system calls compared to just using FileWriter if there are multiple writes before being closed (file io - Java difference between FileWriter and BufferedWriter - Stack Overflow).

ReadFiles

The ReadFiles class contains static methods that read objects from files such as staff, students, and courses. This and the WriteFiles class make up the basis of the record system. It creates a new ArrayList of the type of object it will be reading and then creates a new array of files that is set to the list of files found in the directory for the object type. After checking the array isn’t null, it then iterates through the array and reads each object individually before checking if the object returned is null, if not it then gets added to the global ArrayList for the object type.

Individual students are read using BufferedReader over FileReader comes with multiple benefits, detailed below in the table created by GeeksforGeeks (Difference Between BufferedReader and FileReader in Java - GeeksforGeeks).

Testing

Overview

To test my system, I decided to use some unit tests which are tests that focus on individual parts of the system to check they all work and help isolate areas of the system (Unit Testing - Software Testing - GeeksforGeeks). I have tested some key functions using JUnit 4, an open-source testing framework that uses code annotations to identify test methods to be run (Unit Testing with JUnit 4 - Tutorial (vogella.com)).

The key functions I decided to unit test were the password checks for the login of each type of user and the reading and writing of files for each type of object written to a file. Ideally all functions should be tested in any way possible but in the interest of time, I decided to limit my testing to a few important functions.
